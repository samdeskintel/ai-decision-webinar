07 / 37
Kris
The root cause
GSOCs were built for a world with a detection problem.
The world moved. The model didn't.
When your operation was designed
Detection was the hard problem — finding signals in noise
Alert volume was manageable — analysts could keep up
Context was buildable in a reasonable timeframe
Tools, metrics, shift patterns — all built around detection
Human analysts in the middle made sense at the scale
This model was rational. It was right for its time.
What changed
Volume scaled — 1M+ signals per day, globally
Geographies expanded — more people, more assets, more exposure
Board expectations rose — security as a strategic function
The hard problem shifted: from finding signals to trusting decisions at speed
The model didn't move — you patched it with more people
The gap between the model and the world keeps widening.

09 / 37
Kris
The tool trap
Every tool in your stack was built for a different part of the same wrong problem
Detection tools
Dataminr, Factal and others made signal detection faster. More coverage, better speed, less noise than manual monitoring.

But an alert is not a decision. The work that turns an alert into a trusted action still happens after the tool fires — in the human gap.
Faster signal. The decision is still yours to assemble.
Specialist tools
Everbridge for notification. Ontic for investigations. ESRI for GIS. Each solves a real problem — execution, depth, geography.

None of them connect detection to decision. They're downstream of the problem.
Better execution. The decision gap is still upstream.
The invisible layer
Between detection and execution lives the decision. Validate accuracy. Build context. Map org exposure. Apply threshold. Determine action.

That work was never owned by any tool. It was always human. It still is. It's where the 6 hours live.
No tool was ever built for the middle. Until now.
You bought speed at the front and back of the workflow. The middle — where the decision lives — stayed slow. Because no one named it as the problem.

Kris
The real reason change stalls
Teams don't stay stuck because they lack motivation. They stay stuck because the cost of staying still is invisible.
The cost is structural — not visible
Six hours of analyst time per incident doesn't appear as a line item. It shows up as headcount, contractor spend, and burnout. Nobody connects it to the model. So the model doesn't change.
You can't make a case for change with a cost no one can see.
The current model feels like it's working
Incidents get handled. Notifications go out. Nothing catastrophic happened last quarter. That's enough to defend the status quo — even while the gap widens between what the operation costs and what it produces.
Absence of visible failure is not evidence of operational health.
The improvement metric doesn't exist
If you don't measure time-to-decision, you can't show it improving. If you can't show it improving, you can't make the case for investment. The absence of the metric is what makes the inertia permanent.
The first act of changing the model is deciding to measure it.
Transformation feels like a project
Teams hear "change the operational model" and imagine a 12-month transformation programme. So they defer it. The teams that move fastest discover it doesn't work that way — one stage, measured, then the next.
The fastest path starts with one number, not a roadmap.
